分布式事务的解决方案

一、概述

​ 分布式事务是指涉及多个独立计算节点或数据存储节点的事务处理过程。在分布式计算环境中，由于数据和计算资源分布在不同的地方，因此需要一种机制来确保在多个节点上执行的操作能够保持一致性和可靠性，就像在单一计算节点上执行一样。

​ 简单来说就是多个数据库需要保证一致性，需要事务的支持，但是 MySQL 原生的事务是不支持跨库处理的，所以需要一种新的解决方案。

二、相关概念

ACID 属性

​ 传统的事务处理遵循ACID属性，比如 MySQL，即原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）和持久性（Durability）。在分布式环境中，也需要保证这些属性，但是确保这些属性通常更为复杂。

CAP 理论

​ CAP 定理是分布式系统设计中的一个基本概念，它指出在分布式系统中，一致性（Consistency）、可用性（Availability）、分区容忍性（Partition Tolerance）这三者之间存在权衡关系，不可能同时满足。系统设计者需要根据具体需求选择其中两个属性进行权衡。

​ 因为网络分区的出现，即在保证分区容错性是前提，所以不能保证同时一致性和可用性，所以 CA 并不存在。因此分布式系统理论上只能选择 AP 或者 CP 架构。

例如：

  • ZooKeeper、HBase CP 架构
  • Cassandra、Eureka 就是 AP 架构
  • Nacos 不仅支持 CP 架构也支持 AP 架构

在进行分布式系统设计和开发时，我们不应该仅仅局限在 CAP 问题上，还要关注系统的扩展性、可用性等等，在系统发生“分区”的情况下，CAP 理论只能满足 CP 或者 AP。

BASE 理论

BASE 是 Basically Available（基本可用）、Soft-state（软状态） 和 Eventually Consistent（最终一致性） 三个短语的缩写。BASE 理论是对 CAP 中一致性 C 和可用性 A 权衡的结果，其来源于对大规模互联网系统分布式实践的总结，是基于 CAP 定理逐步演化而来的，它大大降低了我们对系统的要求。

BASE 理论就是牺牲一定的一致性来换取一部分的可用性的策略。

三、分布式事务解决方案

两阶段提交 2PC

第一个阶段是「投票阶段」

  • 1.协调者首先将命令「写入日志」
    1. 「发一个prepare命令」给B和C节点这两个参与者
    1. B和C收到消息后，根据自己的实际情况，「判断自己的实际情况是否可以提交」
  • 4.将处理结果「记录到日志」系统
  • 5.将结果「返回」给协调者

第二个阶段是「决定阶段」

当A节点收到B和C参与者所有的确认消息后

  • 「判断」所有协调者「是否都可以提交」
    • 如果可以则「写入日志」并且发起commit命令
    • 有一个不可以则「写入日志」并且发起abort命令
  • 参与者收到协调者发起的命令，「执行命令」
  • 将执行命令及结果「写入日志」
  • 「返回结果」给协调者

缺点

  • 单点故障
  • 数据不一致
  • 响应时间长
  • 不确定性

三阶段提交 3PC

三阶段提交又称3PC，相对于2PC来说增加了CanCommit阶段和超时机制。如果第三段时间内没有收到协调者的commit请求，那么就会自动进行commit，解决了2PC单点故障的问题。

使用 MQ + 补偿法

  • 通过一个本地消息的数据库表来记录消息具体的发送情况，如果发送失败，可以结合定时调度完成补偿。
  • 如果消息失败了，会根据具体的设计来完成对数据库的回滚，或则再次发送消息进行重试

缺点：

  • 一致性和系统可用性的关系，强调数据一致，会导致定时任务调度过于频繁，系统的可用性损耗。
  • 代码侵入较多，需要手写代码以及设计具体的流程，不易于扩展。

Seata

架构

TC (Transaction Coordinator) -事务协调者：维护全局和分支事务的状态，协调全局事务提交或回滚

TM (Transaction Manager) -事务管理器：定义全局事务的范围、开始全局事务、提交或回滚全局事务

RM (Resource Manager) -资源管理器:管理分支事务处理的资源，与TC交谈以注册分支事务和报告分支事务的状态，并驱动分支事务提交或回滚

XA 模式

![po_diagram (1)](../images/分布式事务的解决方案/po_diagram (1).png)

优点：

  • 事务的强一致性，满足ACID原则
  • 常用数据库都支持，实现简单，并且没有代码侵入

缺点：

  • 一阶段需要锁定数据库资源，等待二阶段结束才释放，性能较差
  • 依赖关系型数据库实现事务

实现方式

修改application.yml文件（每个参与事务的微服务），开启XA模式

给发起全局事务的入口方法添加 @GlobalTransactional

AT 模式

![po_diagram (2)](../images/分布式事务的解决方案/po_diagram (2).png)

优点：

  • 一阶段完成直接提交事务，释放数据库资源，性能比较好
  • 利用全局锁实现读写隔离
  • 没有代码侵入，框架自动完成回滚和提交

缺点：

  • 两阶段之间属于软状态，属于最终一致
  • 框架的快照功能会影响性能，但比XA模式要好很多

TCC 模式

![po_diagram (3)](../images/分布式事务的解决方案/po_diagram (3).png)

优点：

  • 一阶段完成直接提交事务，释放数据库资源，性能好
  • 相比AT模型，无需生成快照，无需使用全局锁，性能最强
  • 不依赖数据库事务，而是依赖补偿操作，可以用于非事务型数据库

缺点:

  • 有代码侵入，需要人为编写try、Confirm和Cancel接口，太麻烦
  • 软状态，事务是最终一致
  • 需要考虑Confirm和Cancel的失败情况，做好幂等处理

​

SAGA 模式

![po_diagram (4)](../images/分布式事务的解决方案/po_diagram (4).png)

优点：

  • 事务参与者可以基于事件驱动实现异步调用，吞吐高
  • 一阶段直接提交事务，无锁，性能好
  • 不用编写TCC中的三个阶段，实现简单

缺点：

  • 软状态持续时间不确定，时效性差
  • 没有锁，没有事务隔离，会有脏写

四、技术对比

一致性保证

  • Seata是一个专门设计用于分布式事务管理的工具，它采用2PC，提供了强一致性的事务支持。
  • MQ补偿法通常使用异步消息队列来处理分布式事务，因此可能会引入一定程度的不一致性。虽然在大多数情况下可以实现一致性，但在极端情况下，消息可能会丢失或消息队列出现故障，导致事务不一致。

性能和可伸缩性

  • Seata采用同步方式，这可能会导致性能瓶颈，尤其是在高并发事务场景下。但它适用于需要强一致性的业务场景。
  • MQ补偿法使用异步消息传递，因此在可伸缩性和性能方面通常更具优势。它可以更好地处理高并发情况，并且可以分散事务负载。

实现复杂性

  • Seata提供了一套完整的分布式事务管理框架，但需要在应用程序中集成Seata的客户端库，这可能需要一些额外的工作。
  • 使用MQ补偿法可能需要更多的自定义代码，因为需要自行实现消息的发送、接收和补偿逻辑。这可能导致一些额外的开发工作和维护负担。

果需要强一致性、可靠性和集成性高的分布式事务解决方案，Seata可能更合适。如果您更注重性能、可伸缩性和更灵活的消息传递，那么MQ补偿法可能更适合您。
